# LLM Proxy — Анализ моделей и промптов

**Дата:** Июнь 2026  
**Контекст:** Образовательный бот «Объяснятель ДЗ», математика 1–4 класс  
**Цель:** Снизить latency с 30–60 с до 10–20 с  
**Ключевой принцип:** Каждый шаг пайплайна имеет разные требования — для каждого можно выбрать оптимального провайдера независимо

## 1. Текущее состояние пайплайна

### Модели (жёстко прошиты, TODO-комментарии)

| Шаг | Файл | Текущая модель | Тип входа | Токены промпта |
|-----|------|---------------|-----------|----------------|
| **detect** | `gpt/detect.go:28` | `gpt-4.1-mini` | vision | system ~1 000 + user ~680 |
| **parse** | `gpt/parse.go:26` | `gpt-4.1-mini` | vision | system ~1 750 + user ~300 |
| **hint** | `gpt/hint.go:27` | `gpt-5-mini` | **text only** | system **~4 400** + user ~520 |
| **check** | `gpt/check.go:33` | `gpt-5-mini` | vision | system **~4 700** + user ~800 |

### Пайплайн Help (фото → подсказки)

```
Фото → detect (gpt-4.1-mini, vision) → parse (gpt-4.1-mini, vision) → hint (gpt-5-mini, text)
```

Все три вызова **строго последовательные** — каждый ждёт предыдущего.

### Пайплайн Check (фото → проверка)

```
Фото → detect (gpt-4.1-mini, vision) → parse (gpt-4.1-mini, vision) → check (gpt-5-mini, vision)
```

### Корневые причины медленного ответа

1. **gpt-5-mini в hint** — text-only задача на мощной модели с высоким TTFB (~15–25 с)
2. **gpt-5-mini в check** — vision + reasoning, обоснованно, но медленно (~10–20 с)
3. **Три последовательных round-trip** к OpenAI API
4. **Огромные system prompt-ы**: hint.system = 17.7 КБ, check.system = 18.7 КБ
5. **Дублирование в detect**: system и user prompt содержат одни и те же правила разными словами

## 2. Анализ промптов

### detect.system (3.9 КБ) + detect.user (2.6 КБ)

**Проблема: дублирование.** System описывает правила quality/classification детально, user.txt повторяет их в другом порядке. User prompt должен содержать только формат входа.

**Оптимизация:** Убрать из user.txt все правила, оставить только:
```
Верни JSON по detect.schema v2.2.2.
Вход: homework_image + optional locale.
```
→ Экономия ~600 токенов на каждый detect-вызов.

---

### parse.system (6.9 КБ)

**Проблема: visual_reasoning всегда обязателен.** Для 80%+ текстовых задач без рисунков модель генерирует пустой visual_reasoning (лишние 50–200 output-токенов).

**Оптимизация:** Изменить правило — заполнять `visual_reasoning` только если в task_text_clean есть слова: «рисунок», «схема», «диаграмма», «весы», «чертёж». Иначе `null`.

Дополнительно: `solution_internal.plan` запрашивает 2–6 шагов и детальные `solution_steps` — это тяжело по output-токенам. Можно сократить до max 4 шагов plan + краткие steps.

---

### hint.system (17.7 КБ) — самый проблемный

**Проблема:** Гигантский промпт с правилами для edge-кейсов (олимпиадные задачи, профили шаблонов, visual_facts_required_gate, absurd_result_gate). Типичный вызов — простая задача 2 класса без шаблонных профилей.

**Оптимизация:** Разделить на:
- **hint.base.system** (~3 КБ) — базовые правила для 90% случаев
- **hint.advanced.system** (~8 КБ) — подключается только при наличии шаблонного профиля или сложных флагов

Примерная экономия: 3 000–4 000 input-токенов = ~1–2 с быстрее + дешевле.

---

### check.system (18.7 КБ)

**Проблема:** Обоснованно детальный для сложных задач, но для простой арифметики (12+12=?) это overhead. Секции 3.3.1 (BFS-алгоритмы, перебор возрастов) нужны редко.

**Оптимизация (осторожно):** Можно вынести секцию 3.3.1 в отдельный `check.advanced.system`, подключаемый по ped_keys.constraints.

## 3. Рекомендации по моделям (текущие изменения)

### P0 — одна строка, максимальный эффект

**hint.go:27** → заменить `gpt-5-mini` на `gpt-4.1-mini`

- Hint — **чисто текстовая задача**, никакого изображения
- Форматирование педагогических подсказок на основе готового JSON из parse
- gpt-5-mini не даёт ощутимого преимущества для 1–4 класса
- **Ожидаемый выигрыш: −15–20 с**

### P1 — среднее влияние на качество

**check.go:33** → `gpt-5-mini` на `gpt-4.1-mini`

- Проверка арифметики 1–4 класса доступна 4.1-mini
- Если обнаружится деградация на сложных задачах — вернуть gpt-5-mini только для `high_risk`
- **Ожидаемый выигрыш: −5–10 с**

### P2 — архитектурное изменение

Объединить **detect + parse** в один вызов:
- Detect делает лишь 2 простые вещи (quality check + subject) и его результат нужен только parse
- Одна vision-задача вместо двух = минус один round-trip
- **Ожидаемый выигрыш: −3–5 с**

## 4. OpenRouter — единая точка доступа ко всем провайдерам

### Что такое OpenRouter

[OpenRouter](https://openrouter.ai/) — прокси-агрегатор, который предоставляет **один OpenAI-совместимый API** для 300+ моделей от всех крупных провайдеров: OpenAI, Anthropic, Google, Meta, Qwen, Mistral и других.

```
Наш LLM Proxy → OpenRouter API → OpenAI / Gemini / Claude / Qwen / ...
```

Вместо реализации отдельных клиентов под каждого провайдера — меняем только **строку с именем модели**.

---

### Как это меняет архитектуру

**Сейчас:** один провайдер (OpenAI), код жёстко вызывает `api.openai.com/v1/responses`

**С OpenRouter:** один клиент, менять провайдера = менять строку модели:

```go
// detect.go — было:
model = "gpt-4.1-mini"
// стало:
model = "google/gemini-2.0-flash-lite"

// hint.go — было:
model = "gpt-5-mini"
// стало:
model = "openai/gpt-4.1-mini"
// или:
model = "anthropic/claude-haiku-4-5"
```

**Важно:** текущий код использует **OpenAI Responses API** (`/v1/responses`), а OpenRouter — стандартный **Chat Completions API** (`/v1/chat/completions`). Формат тела запроса нужно адаптировать (поле `messages` вместо `input`, формат Vision-вложений — аналогичный).

---

### Что поддерживает OpenRouter

| Фича | Статус |
|------|--------|
| Vision (изображения) | ✅ для большинства мультимодальных моделей |
| Structured Output / JSON Schema | ✅ для OpenAI, Gemini, частично Anthropic |
| Streaming | ✅ |
| Provider routing (приоритет провайдера) | ✅ можно указать `provider.order` |
| Автоматический fallback | ✅ при ошибке переключается на следующего провайдера |
| Балансировка нагрузки | ✅ |
| Единый биллинг | ✅ один баланс для всех провайдеров |

### Доступность из России

OpenRouter — американский сервис. Доступ из России:
- Прямые запросы из РФ часто блокируются
- **Решение:** те же методы, что уже используются для OpenAI — VPN/proxy на стороне сервера (llm_proxy уже работает через `http.ProxyFromEnvironment`)
- Один прокси-сервер закрывает доступ ко всем провайдерам через OpenRouter


**Стоимость /1к Help-сессий через OpenRouter:** ~$1.80–$2.10 (с учётом +100ms latency overhead)

### Что нужно изменить в коде

1. **Формат запроса:** `input` → `messages`, `text.format` → `response_format`

```go
// Было (OpenAI Responses API):
body := map[string]any{
    "model": model,
    "input": []any{ ... },
    "text": map[string]any{
        "format": map[string]any{"type": "json_schema", ...},
    },
}

// Стало (Chat Completions API — OpenRouter):
body := map[string]any{
    "model": model,
    "messages": []any{ ... },
    "response_format": map[string]any{
        "type": "json_schema",
        "json_schema": map[string]any{"name": name, "strict": true, "schema": schema},
    },
}
```

2. **Vision-вложения:** формат data URL остаётся таким же (`image_url.url = "data:image/jpeg;base64,..."`)

3. **Заголовки:**

```go
req.Header.Set("Authorization", "Bearer "+openRouterAPIKey)
req.Header.Set("HTTP-Referer", "https://vk.obyasnyatel.ru")  // опционально
req.Header.Set("X-Title", "Объяснятель ДЗ")                  // опционально
```

4. **Парсинг ответа:** стандартный Chat Completions формат (`.choices[0].message.content`)

## 6. Стратегия смешанных провайдеров

Каждый шаг пайплайна имеет принципиально разные требования:

| Шаг | Что важно | Что НЕ важно |
|-----|-----------|--------------|
| **detect** | Скорость, дешевизна, базовое vision | Точное OCR, сложный reasoning |
| **parse** | Точное OCR (особенно рукопись), JSON-структура, русский | Скорость (узкое место всё равно здесь) |
| **hint** | Качество русского текста, строгий JSON, стиль для детей | Vision (нет изображения!) |
| **check** | Vision + математический reasoning | Скорость генерации текста |

Архитектура `engine.go` абстрагирует провайдера — можно добавить Gemini/Claude клиент рядом с GPT-клиентом.

#### detect — что использовать

**Задача:** оценить читаемость фото + определить предмет (math/ru/en/other).  
Простая бинарная + 6-way классификация. Ошибка в detect = пользователь видит "переснять" — некритично.

| **gpt-4.1-nano** | Дешевле mini в 4x, быстрый TTFB ~0.5 с | Чуть слабее на сложных случаях | ✅ Лучший для detect |
| **Gemini 2.0 Flash Lite** | Дешевле всех ($0.075/M), 0.4 с TTFB | Требует Gemini-клиент | ✅ Если уже есть Gemini |
| gpt-4.1-mini (текущий) | Надёжный | Переплата в 4x vs nano | ⚠️ Можно заменить |

| **Qwen2.5-VL-72B** | Лучший OCR документов на рынке | Latency ~1–3 с (география), JSON нестрогий | ⚠️ Тест перед production |

**Задача:** смотреть на фото ответа ученика + проверять математику + давать обратную связь.  
Нужны: vision + математические рассуждения. Чувствительный к ошибкам шаг.

| Кандидат | Плюсы | Минусы | Рекомендация |
|----------|-------|--------|--------------|
| **gpt-4.1-mini** | Надёжный, хороший OCR ответа, дешевле 5-mini в 3x | Чуть слабее на high-risk | ✅ Основной кандидат |

### Что НЕ делать

- **Не заменять parse на nano** — OCR + планирование решения требует сильной модели
- **Не переходить на Qwen в production** без длительного теста на реальных задачах (JSON нестабильность)
- **Не использовать gpt-5 full** — никакой практической разницы для 1–4 класса при цене в 25x
- **Не объединять detect+parse** без A/B теста — риск потери в качестве parse при экономии одного round-trip
